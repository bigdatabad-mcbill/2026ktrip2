# 2026ktrip2
아래는 **클로즈트립(CloseTrip)**의 실제 개발 가능한 수준을 목표로 한 서비스 UX·기술·사업·론칭 통합 설계안입니다. 외국인도 즉시 이해할 수 있도록 **다국어·다문화 UX를 서비스의 중심 구조로** 두고, MVP에서 검증 가능한 범위와 고도화 범위를 분리해 제안합니다.

***

## 1) 서비스 개요

**CloseTrip**은 “오늘 어디 가고, 무엇을 입고, 어디서 찍고, 어떻게 남길지”를 한 흐름으로 연결하는 여행 앱입니다.  
핵심 가치는 **날씨 기반 코디 추천 + 장소 무드 + 포토스폿 + 카메라 가이드 + 자동 여행 다이어리**를 하나의 경험으로 묶는 데 있습니다.

### 서비스 한 줄 정의
- 여행자의 현재 위치와 일정, 날씨, 장소 특성, 사진 기록을 결합해 **즉시 행동 가능한 여행 의사결정 도구**를 제공하는 앱.
- 단순 정보 제공이 아니라, 사용자가 바로 “오늘 갈 곳/입을 옷/찍을 위치/남길 기록”을 결정하게 만드는 것이 목표.

### 핵심 가치
- **즉시성:** 앱을 열고 3분 안에 “오늘의 여행 플랜”을 이해.
- **신뢰성:** 추천마다 이유를 보여줘 “왜 이걸 추천하는지” 납득 가능.
- **현지성:** 한국어 UI를 단순 번역하는 것이 아니라, 언어·단위·문화적 표현까지 맞춤화.
- **기록 자동화:** 사용자가 따로 정리하지 않아도 여행 다이어리가 자동 생성.

### 서비스 구조 요약
1. **홈**에서 오늘의 여행 상태를 요약.
2. **코디**가 날씨·TPO·활동에 맞춰 제안.
3. **포토스폿**이 지도와 구도로 연결.
4. **카메라**가 촬영 순간을 지원.
5. **다이어리**가 일정·사진·메타데이터로 자동 완성.
6. **설정**에서 언어, 권한, 저장 방식, 개인정보를 통제.

## 2) 타깃 사용자 정의

### 1차 타깃
- **20~39세 국내 여행자**
  - 주말 나들이, 데이트, 미식/카페 투어, 사진 중심 여행.
  - “뭘 입고 어디서 찍을지”에 민감.
- **한국 방문 외국인 관광객**
  - 서울/부산/제주 등 도시 여행자.
  - 언어 장벽이 낮아야 하고, 지리·날씨·결제·교통 이해가 쉬워야 함.

### 2차 타깃
- **여행 콘텐츠 제작자**
  - 인스타/틱톡용 촬영 포인트, 구도, 장소 분위기 필요.
- **패션에 관심 많은 사용자**
  - 여행 코디를 기록하고 재사용하고 싶은 사용자.
- **가벼운 기록 선호 사용자**
  - 사진만 남겨도 자동으로 여행 다이어리가 정리되길 원하는 사용자.

### 페르소나 예시
- **국내 사용자 A**
  - 서울에서 친구와 카페 투어.
  - 날씨가 추우면 코트/신발 추천이 중요.
  - 인생샷 포인트를 지도에서 빠르게 찾고 싶음.
- **외국인 사용자 B**
  - 첫 한국 여행, 한국어를 거의 모름.
  - 언어 선택이 초기에 바로 되어야 함.
  - “근처 인기 장소”와 “사진 찍기 좋은 위치”를 직관적으로 보고 싶음.
- **여행 기록형 사용자 C**
  - 일정은 대충 잡지만 기록은 남기고 싶음.
  - 자동 다이어리와 요약이 핵심 가치.

## 3) 핵심 사용자 시나리오

### 시나리오 A: 첫 방문 외국인 관광객
1. 앱 실행 후 언어 선택 화면 진입.
2. 국가/여행 목적/관심사 선택.
3. 현재 위치 또는 여행 예정 지역 선택.
4. “오늘의 여행 카드”에서 날씨, 코디, 포토스폿, 저장 기능을 한 번에 확인.
5. 지도에서 포토스폿을 누르고 길찾기.
6. 카메라에서 촬영 가이드 확인 후 사진 촬영.
7. 저녁에 자동 생성된 여행 다이어리 확인.

### 시나리오 B: 국내 주말 나들이 사용자
1. 홈에서 오늘 날씨와 추천 장소 확인.
2. “가벼운 산책 / 카페 / 전시” 같은 무드 기반 추천 확인.
3. 옷차림 추천 확인 후 외출.
4. 촬영 포인트에서 가이드라인을 보고 사진 촬영.
5. 여행 종료 후 자동 다이어리로 공유.

### 시나리오 C: 기록 중심 사용자
1. 앱에서 일정을 연결하거나 수동으로 입력.
2. 방문한 장소와 사진이 자동 수집.
3. 다이어리 편집 화면에서 짧은 메모만 입력.
4. 자동 생성된 하루 기록을 SNS용으로 내보내기.

***

### 4-1. 온보딩 UX
**목표:** 3분 안에 핵심 가치 전달.  
**원칙:** 텍스트 최소화, 선택형 중심, 언어 먼저, 권한은 나중.

#### 화면 흐름
1. 언어 선택
2. 여행 목적 선택
3. 관심사 선택
4. 위치/일정 연동 선택
5. 권한 안내
6. 첫 홈 진입

#### 온보딩 문구 예시
- 한국어: “오늘의 여행을 더 쉽게 시작해보세요.”
- 영어: “Plan your trip in one flow.”

- 한국어: “언어를 먼저 선택하세요.”
- 영어: “Choose your language.”

- 한국어: “여행 스타일을 골라주세요.”
- 영어: “Pick your travel style.”

- 한국어: “위치와 사진 권한은 나중에 바꿀 수 있어요.”
- 영어: “You can change location and photo permissions later.”

#### 권한 요청 시점
- 앱 첫 실행에서 즉시 강제 요청하지 않음.
- **가치 경험 후** 권한 요청:
  - 위치: “근처 포토스폿을 보여드리려면 위치가 필요해요.”
  - 사진: “다이어리를 자동으로 만들려면 사진 접근이 필요해요.”

***

### 4-2. 홈 화면
**역할:** 오늘의 핵심 의사결정 허브.  
사용자는 이 화면에서 “오늘 어디 갈지, 무엇을 입을지, 어디서 찍을지”를 즉시 이해해야 합니다.

#### 구성
- 상단: 위치, 날짜, 날씨, 언어
- 메인 카드: Today Trip Summary
- 코디 카드: 기온/강수/활동 기반 추천
- 장소 카드: 추천 여행지 3개
- 포토스폿 카드: 지도 요약 + 거리
- 다이어리 카드: 오늘 기록 상태
- 하단 CTA: “Start Trip”, “Open Camera”, “Save Diary”

#### 홈 문구 예시
- 한국어: “오늘 서울, 18°C, 흐림”
- 영어: “Seoul today, 18°C, cloudy.”

- 한국어: “추천 코디: 가벼운 코트 + 운동화”
- 영어: “Suggested outfit: light coat + sneakers.”

- 한국어: “오늘 찍기 좋은 장소 3곳”
- 영어: “3 spots worth shooting today.”

- 한국어: “여행 기록이 아직 없어요.”
- 영어: “No trip log yet.”

#### 빈 상태 문구
- 한국어: “일정을 추가하면 더 정확하게 추천할 수 있어요.”
- 영어: “Add a trip plan for better recommendations.”

***

### 4-3. 일정(Trip) 화면
**역할:** 여행 계획과 방문 기록을 연결.  
일정은 수동 입력뿐 아니라 캘린더, 지도, 사진 기록과 연동됩니다.

#### 구성
- 날짜별 카드형 일정
- 장소명, 예상 체류 시간, 이동 시간
- 날씨 변화 표시
- “이 일정에 맞는 코디 보기”
- “이 일정의 포토스폿 보기”

#### 라벨 예시
- 한국어: “일정 추가”
- 영어: “Add stop”

- 한국어: “이동 경로”
- 영어: “Route”

- 한국어: “날씨 반영”
- 영어: “Weather-aware”

#### 에러 문구
- 한국어: “장소를 불러올 수 없어요. 잠시 후 다시 시도해주세요.”
- 영어: “We couldn’t load this place. Please try again.”

***

### 4-4. 코디 추천 화면
**역할:** 날씨 + 장소 무드 + 활동 강도를 결합한 추천 제공.  
이 화면은 단순 옷 추천이 아니라, **TPO 기반 스타일링 의사결정**을 돕습니다.

#### 구성
- 상단: 날씨 요약
- 중앙: 추천 코디 카드 3개
- 하단: 코디 근거
  - 기온
  - 강수 가능성
  - 바람
  - 이동량
  - 장소 분위기
- “이 코디를 저장” 버튼
- “비슷한 코디 보기” 버튼

#### 설명 예시
- 한국어: “기온 18°C, 바람이 강해 얇은 겉옷이 적합해요.”
- 영어: “18°C with strong wind, so a light outer layer works best.”

#### 신뢰성 UI
추천 카드 하단에 “Why this?”를 노출:
- “낮 기온과 저녁 기온 차이가 큼”
- “비 예보가 있어 방수 신발 추천”
- “전시 관람 중심 일정이라 편안한 스타일 우선”

***

### 4-5. 포토스폿 화면
**역할:** 지도 기반 촬영 포인트 추천.  
단순 지도 마커가 아니라, “왜 여기서 찍어야 하는지”를 보여줘야 합니다.

#### 구성
- 지도 뷰 + 리스트 뷰 전환
- 포토스폿 카드:
  - 장소명
  - 도보 거리
  - 혼잡도
  - 추천 시간대
  - 대표 구도
- 필터:
  - 실내/실외
  - 사람 적은 곳
  - 야경
  - 비 오는 날
  - 외국인 인기

#### 카드 문구 예시
- 한국어: “해질 무렵에 가장 예뻐요.”
- 영어: “Best at sunset.”

- 한국어: “사람이 적은 시간대”
- 영어: “Less crowded time”

#### 지도 UX
- 현재 위치와 포토스폿의 관계를 한눈에 표시
- 거리 단위를 사용자 언어/지역에 따라 km, m, ft, mi로 전환
- 외국인은 대중교통/도보 기준이 함께 보여야 함

***

### 4-6. 카메라 화면
**역할:** 촬영 순간 가이드 + 기록 자동화.  
카메라는 단순 촬영이 아니라, 포토스폿 경험의 완성 단계입니다.

#### 구성
- 촬영 화면
- 구도 가이드 오버레이
- 수평선 안내
- 인물 위치 프레임
- 추천 필터
- 촬영 후 자동 태그 제안

#### 기능
- AR 오버레이로 촬영 위치 제안
- “여기 서세요” 가이드
- “인물은 왼쪽 1/3” 같은 구도 설명
- 촬영 후 장소명, 시간, 날씨 자동 저장

#### 카피 예시
- 한국어: “이 위치에서 찍으면 배경이 더 잘 보여요.”
- 영어: “This spot gives you a better background.”

***

### 4-7. 다이어리 화면
**역할:** 자동으로 여행 기록을 완성하고, 사용자가 가볍게 편집.  

#### 구성
- 날짜별 여행 카드
- 방문 장소, 사진, 날씨, 이동 경로 요약
- 자동 생성 문장 + 사용자 메모 입력
- 공유용 템플릿
- PDF/이미지 내보내기

#### 자동 생성 예시
- 한국어: “오늘은 비가 왔지만, 오후에는 맑아져서 카페와 전시를 둘러봤어요.”
- 영어: “It rained in the morning, then cleared up in the afternoon for a café and gallery walk.”

#### 빈 상태 문구
- 한국어: “기록할 여행이 아직 없어요. 방문 장소와 사진이 쌓이면 자동으로 만들어져요.”
- 영어: “No trip log yet. Your visits and photos will build it automatically.”

***

### 4-8. 설정 화면
**역할:** 언어, 권한, 단위, 개인정보, 알림 관리.

#### 반드시 포함할 항목
- 언어 선택
- 국가/지역 설정
- 시간 형식 12/24시간
- 온도 단위 C/F
- 거리 단위 km/mi
- 사진 권한
- 위치 권한
- 데이터 보관 정책
- 개인정보 삭제
- 오프라인 모드

#### 다국어 선택 위치
- 설정에만 숨기지 않고 **첫 실행에서 즉시 선택**
- 프로필에서도 빠르게 변경 가능

## 5) 다국어/현지화 정책

### 언어 우선순위
1. 한국어
2. 영어
3. 일본어
4. 중국어 간체
5. 중국어 번체
6. 태국어
7. 베트남어

### 현지화 원칙
- 번역이 아니라 **행동 방식까지 현지화**
- 같은 기능이라도 국가별로 버튼 순서, 예시 이미지, 톤을 조정
- 텍스트 길이 증가를 고려해 버튼과 카드 레이아웃은 유동형으로 설계
- 날짜/시간/단위/통화는 시스템 locale과 사용자 선택을 함께 반영

### LTR/RTL 대응
- 기본은 LTR 중심이지만 다국어 문자열 길이와 줄바꿈 최적화 필요
- 향후 RTL 언어가 추가될 수 있으므로:
  - 텍스트 방향 분리
  - 아이콘 방향 자동 미러링
  - 레이아웃 좌우 고정 금지

### 문화적 표현 차이
- 한국어의 짧고 맥락적인 표현은 영어에서 직접 번역하면 딱딱해질 수 있음.
- 예:
  - 한국어: “지금 가기 좋아요.”
  - 영어: “Best to visit now.”
- 일본어는 정중함, 영어는 명확성, 태국어는 친근함을 기본 톤으로 조정.

### 이미지 톤 정책
- 국가별 대표 이미지와 스타일을 다르게 사용
- 외국인용 이미지는:
  - 한국 특유의 장소성은 유지
  - 과도한 로컬 농담, 은어, 너무 복잡한 UI 문구는 배제
- 장소 추천 시 배경 이미지가 사용자의 성별/연령 고정관념을 강화하지 않도록 주의

### 문구 예시
| 기능 | 한국어 | 영어 |
|---|---|---|
| 언어 선택 | 언어를 선택하세요 | Choose your language |
| 코디 추천 | 오늘 입기 좋아요 | Good to wear today |
| 포토스폿 | 사진 찍기 좋은 곳 | Great photo spot |
| 다이어리 | 오늘 여행을 자동으로 정리했어요 | Your trip is now auto-organized |

## 6) 기술 아키텍처

### 전체 구조
- **Client App**
  - iOS / Android
  - React Native 또는 Flutter 권장
- **API Gateway**
  - 인증, 라우팅, rate limit
- **Core Services**
  - User Service

  - Place Service
  - Recommendation Service
  - Diary Service
  - Notification Service
  - Translation/Localization Service
- **Data Layer**
  - PostgreSQL
  - Redis

  - Search/Geo Index(Elasticsearch 또는 OpenSearch)
- **AI Layer**
  - 추천 규칙 엔진 + 경량 ML/LLM 보조
  - 이미지 태깅, 요약 생성, 캡션 생성
- **Analytics**
  - 이벤트 수집, 퍼널 분석, A/B 테스트

### 데이터 연결 방식
#### 1. 날씨 API
- 도시/좌표/시간대 기준 조회
- 온도, 강수, 풍속, 체감온도, 자외선 등을 저장
- 추천에 필요한 feature로 변환

#### 2. 일정 데이터
- 사용자의 여행 날짜, 장소, 체류 시간, 이동 경로를 저장
- 캘린더 연동 가능
- 장소 방문 로그와 결합

#### 3. 장소 데이터
- POI, 영업시간, 카테고리, 리뷰, 혼잡도, 사진 포인트
- 지도 SDK와 연결
- 외부 장소 API 의존 시 캐시 레이어 필수

#### 4. 스타일 추천 로직
- 입력: 날씨, 장소 유형, 이동량, 사용자 선호, 과거 저장 코디
- 출력: 코디 3안 + 이유 + 대체안
- 초기는 룰 기반이 적합, 이후 개인화 모델 도입

### 추천 엔진 구현 방식
#### MVP
- Rule-based + scoring
- 예:
  - 강수 확률 높음 → 방수 재킷 가중치 상승
  - 이동량 많음 → 편한 신발 가중치 상승
  - 야간 촬영 중심 → 외투/보온 가중치 상승

#### 고도화
- 사용자 선호 학습
- 계절/지역/활동 패턴 기반 개인화
- 이미지 스타일 분석으로 저장 코디와 유사 코디 추천

### 자동 다이어리 생성
- 입력:
  - 사진 메타데이터(EXIF)
  - 위치 기록
  - 일정 정보
  - 날씨 타임라인
- 처리:
  - 하루 요약 생성
  - 장소 순서 정리
  - 대표 사진 선정
  - 짧은 텍스트 생성
- 출력:
  - 일기 카드
  - SNS 공유 템플릿
  - PDF 내보내기

### 외부 의존도 낮추는 대체 방안
- 날씨 API 장애 시:
  - 최근 캐시
  - 동일 도시 평균값
  - 사용자 수동 입력
- 장소 API 장애 시:
  - 자체 POI DB 캐시
  - 인기 장소 우선 제공
- 번역 API 장애 시:
  - 정적 번역 리소스 fallback
- 지도 API 장애 시:
  - 리스트 우선 모드 제공

### 권장 기술 스택
- **Frontend:** Flutter 또는 React Native
- **Backend:** Node.js/NestJS 또는 Kotlin/Spring Boot
- **DB:** PostgreSQL + Redis
- **Storage:** S3 호환 객체 저장소
- **Search/Geo:** OpenSearch
- **Maps:** Google Maps / Naver Map / Kakao Map 멀티 어댑터
- **Weather:** OpenWeather, Tomorrow.io, 또는 지역별 대체 API
- **Analytics:** Amplitude/Firebase + 자체 이벤트 파이프라인
- **Auth:** Apple/Google/Kakao + 이메일
- **AI:** 서버사이드 요약/추천, 온디바이스 일부 처리 가능

### 개인정보/보안 리스크 대응
- 위치정보는 **명시적 동의 기반**
- 사진 접근은 앨범 전체가 아닌 선택 접근 우선
- 민감한 위치 이력은 기본 비공개
- 데이터 보관 기간 설정 가능
- 여행 종료 후 자동 아카이브/삭제 옵션 제공
- 외국인 사용자 데이터는 국가별 법규 준수 고려
- 익명화된 분석 데이터만 서비스 개선에 활용

## 8) 비즈니스 모델

### 모델 비교
| 모델 | 장점 | 단점 | 적합 시점 |
|---|---|---|---|
| 패션 커머스 제휴 | 코디 추천과 자연스럽게 연결 | 추천 신뢰성 훼손 위험 | 초기~중기 |
| 지역 관광 제휴 | 장소/입장권/투어와 연결 가능 | 지역별 운영 복잡 | 중기 이후 |
| 프리미엄 구독 | 광고 없이 고급 기능 제공 가능 | 초기 전환률 낮을 수 있음 | 기능 성숙 후 |
| B2B 데이터/광고 | 확장성 큼 | 개인정보/신뢰 이슈 | 후반부 |

### 우선순위 제안
**초기에는 패션 커머스 제휴 + 지역 관광 제휴의 얇은 형태**를 우선 추천합니다.  
이유는 사용자의 핵심 가치인 “코디”와 “포토스폿”이 바로 구매/예약으로 이어질 수 있기 때문입니다.  
다만 초반에는 광고처럼 보이지 않게, **“추천 근거 기반의 자연스러운 링크”**로만 노출해야 합니다.

### 프리미엄 구독 예시
- 여행 다이어리 고급 템플릿
- 무제한 자동 정리
- 고급 코디 추천
- 고해상도 내보내기
- 여행 데이터 백업

### 충돌 방지 원칙
- 수익화는 추천 품질을 해치면 안 됨
- 제휴 상품은 추천 근거와 분리해서 명시
- “광고”보다 “대안” 형태로 제공
- 사용자가 수익화 노출을 줄일 수 있게 설정 제공

## 10) 심사위원 Q&A 대응 논리

### Q1. 왜 이 서비스가 필요한가?
**답변 논리:**  
여행자는 늘 “오늘 어디 갈지, 무엇을 입을지, 어디서 찍을지, 어떻게 남길지”를 따로 판단해야 합니다. CloseTrip은 이 결정을 하나의 흐름으로 묶어 여행 준비와 실행 비용을 줄입니다.

### Q2. 기존 날씨 앱, 지도 앱, 다이어리 앱과 뭐가 다른가?
**답변 논리:**  
기존 서비스는 기능이 분리돼 있습니다. CloseTrip은 날씨, 장소, 코디, 촬영, 기록을 **연결**하기 때문에 사용자의 실제 행동을 더 잘 지원합니다.

### Q3. 추천이 왜 신뢰할 만한가?
**답변 논리:**  
모든 추천에 “왜 이 추천인지”를 보여주고, 날씨·이동량·장소 특성·사용자 선호를 함께 노출합니다. 초기에는 복잡한 AI보다 규칙 기반으로 시작해 설명 가능성을 우선합니다.

### Q4. 외국인은 정말 쉽게 쓸 수 있는가?
**답변 논리:**  
언어를 설정 메뉴에 숨기지 않고 첫 실행에서 바로 선택하게 합니다. 또한 단위, 날짜, 시간, 문구 톤, 이미지까지 현지화해 단순 번역 앱과 다르게 설계합니다.

### Q5. AI를 과도하게 쓰는 것 아닌가?
**답변 논리:**  
핵심 UX는 AI가 아니라 구조입니다. AI는 다이어리 요약, 추천 보조, 태깅 등에 제한적으로 쓰고, 판단이 필요한 영역은 규칙 기반과 근거 표시로 통제합니다.

### Q6. 구현 난도가 너무 높지 않나?
**답변 논리:**  
MVP에서는 기능을 다 붙이지 않고, 날씨-코디-포토스폿-기록의 최소 루프만 만듭니다. AR, 고도화 개인화, 자동 일정 생성은 2차 기능으로 분리해 기술 리스크를 낮춥니다.

### Q7. 수익화가 UX를 해치지 않나?
**답변 논리:**  
초기 수익화는 광고보다 추천 기반 제휴로 설계합니다. 사용자가 얻는 가치와 바로 연결되는 지점만 유료화해 경험 훼손을 최소화합니다.

### Q8. 데이터/개인정보 리스크는?
**답변 논리:**  
위치와 사진은 명시적 동의 기반으로 처리하고, 기본값은 최소 수집입니다. 여행 종료 후 자동 삭제/아카이브, 익명 분석, 권한 세분화로 리스크를 줄입니다.
